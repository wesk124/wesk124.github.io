Building a RAG-Based Multi-Agent Chatbot

Sheng Wei · February 14, 2026 · 8 min read

For my EEP 596 course at UW, I built a multi-agent chatbot designed to answer questions about machine learning concepts. This post covers the architecture decisions, what worked, and what I'd do differently.

Why Multi-Agent?

A naive RAG pipeline — embed query, retrieve chunks, stuff into context, generate — works surprisingly well for simple factual questions. But it breaks down on complex queries that require multiple retrieval steps or reasoning over the retrieved content.

Multi-agent architectures address this by decomposing the problem: one agent handles query understanding and decomposition, another manages retrieval, and a third synthesizes the final response. The coordination overhead is real, but the quality improvement on hard questions justifies it.

Architecture Overview

The system has three main components:

  • Query Decomposer. Takes a user question and breaks it into sub-queries that can each be answered independently by the retriever.
  • Retrieval Agent. For each sub-query, embeds the query and performs approximate nearest-neighbor search over a pre-built vector index of ML textbook content.
  • Synthesis Agent. Takes the retrieved chunks and sub-answers, reasons over them, and produces a coherent final response with citations.

Vector Store and Embeddings

I indexed roughly 2,000 chunks from several ML textbooks and lecture notes using sentence-transformers (all-MiniLM-L6-v2) and stored them in a FAISS index. Chunk size was 512 tokens with a 64-token overlap — tuned empirically by measuring retrieval recall on a small labeled eval set. 

from sentence_transformers import SentenceTransformer
import faiss, numpy as np

model = SentenceTransformer("all-MiniLM-L6-v2")
embeddings = model.encode(chunks, batch_size=64, show_progress_bar=True)
embeddings = np.array(embeddings).astype("float32")

index = faiss.IndexFlatIP(embeddings.shape[1])
faiss.normalize_L2(embeddings)
index.add(embeddings)

Key Lessons

Retrieval quality is the bottleneck. Spending time on chunking strategy and embedding model selection had a bigger payoff than tuning the generation prompt.

Agent communication needs structure. Passing free-form text between agents led to fragile behavior. Switching to typed JSON messages with explicit fields made the pipeline much more reliable.

Latency compounds. Three sequential LLM calls add up fast. For a production system, I'd parallelize the sub-query retrievals and look at smaller, fine-tuned models for the decomposition step.

What I'd Do Differently

The biggest gap in the current system is evaluation. I relied heavily on subjective quality judgments. A proper eval harness with a labeled question-answer set would have let me iterate faster and make principled decisions about architecture changes.

I'd also explore hybrid retrieval — combining dense embeddings with BM25 sparse retrieval — which consistently outperforms pure dense retrieval on domain-specific corpora.
